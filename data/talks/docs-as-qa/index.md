summary: |
  This talk explores how technical writing naturally functions as quality assurance. We start by showing that the qualities of good QA — bug catching, user experience sense-checks, customer focus, coverage of key paths — are the same qualities that define good technical writing. Technical writers wear many hats and their role extends well beyond documenting existing products; they provide valuable feedback on usability and design.

  Using real examples from Codat's rate limits documentation, we show how the process of writing docs surfaces product issues: when documentation is hard to write, something stinks. We cover "bad smells" — signs like excessive content length, needing too much contextual explanation, content that doesn't fit the target persona, or difficulty creating screenshots — that signal underlying product problems. When writers spot these issues, they should speak up.

  We make the case for shifting left: involving technical writers earlier in the development lifecycle, since defects found earlier are cheaper to fix. This means embedding writers in teams, partnering them with PMs, and even adopting documentation-first approaches like OpenAPI specs as contracts and document-driven development. QA and technical writing naturally complement each other — test plans make excellent starting points for documentation.
